## 글쓰기 스타일 가이드

참고 자료: toss.tech 블로그 (Harness, AI 대체, 실험 설계 아티클)

### 문장 규칙

- **짧고 끊어서** 쓴다. 한 문장에 하나의 생각만.
- 긴 설명이 필요하면 문장을 나눈다. 쉼표로 잇지 않는다.
- "~입니다", "~합니다" 문어체를 기본으로 한다.
- 구어체("~거든요", "~잖아요")는 쓰지 않는다.

### 단락 규칙

- 단락은 3~4문장 이하로 짧게 끊는다.
- 한 단락 = 하나의 주제.
- 단락 사이 빈 줄은 반드시 삽입.

### 소제목 패턴

질문형과 핵심어형을 혼용한다.

```
## 당신의 팀은 같은 LLM을 쓰고 있나요?        ← 질문형
## Executable SSOT: 문서는 죽고, 코드는 산다  ← 핵심어형
## 왜 oh-my-claudecode를 선택하지 않았나       ← 판단형
```

### 강조 방식

- **굵게**: 단락에서 가장 핵심이 되는 한 문장만 굵게.
- 인용구(`>`): 철학적 주장, 핵심 원칙을 한 줄로 요약할 때.
- 코드 블록: 구조, 흐름, 명령어 표시.

```markdown
**차이는 단 하나, 작업 전에 컨텍스트를 설계했느냐의 여부입니다.**

> 생성은 GPT, 판단은 Claude, 통제는 내가 한다.
```

### 논리 전개 구조

```
문제 제기 → 반전("하지만", "그러나") → 분석 → 결론
```

- 결론을 먼저 말하고 근거를 붙이는 것도 허용.
- 반전 없이 단선으로 전개하면 글이 밋밋해진다.

### 비유와 데이터 사용

- 낯선 개념은 익숙한 것으로 비유한다.
  - "oh-my-zsh가 터미널 생산성의 표준이 되었듯"
  - "1995년 닷컴 버블처럼"
- 주장에는 구체적 수치를 붙인다.
  - "5,600억 달러 투입, 실제 매출 350억 달러"
  - "전환율 5% 상승"

### 목록 사용 기준

복잡한 내용, 3개 이상의 항목은 리스트로 정리한다.
단순 나열이 아니라 **각 항목에 이유나 설명**을 붙인다.

```markdown
- **Global Layer**: 전사 공통 규정 (보안 정책, 기본 코딩 스타일)
- **Domain Layer**: 팀/비즈니스별 지식 (특정 도메인 로직)
- **Local Layer**: 특정 레포지토리의 구현 디테일
```

### 마무리 방식

- 핵심 메시지를 1~3문장으로 압축해서 끝낸다.
- "~를 향해", "~가 중요하다" 형식의 선언으로 마무리.
- 독자에게 행동을 요구하거나 질문을 남겨도 된다.

### 금지 패턴

| 하지 말 것 | 이유 |
| --- | --- |
| 긴 문장 쉼표로 연결 | 읽기 어려움 |
| "~인 것 같습니다" | 자신감 없어 보임 |
| 소제목 없이 긴 본문 | 가독성 저하 |
| 서론이 너무 긴 글 | 독자가 이탈 |
| 결론 없이 나열만 | 메시지 없음 |
